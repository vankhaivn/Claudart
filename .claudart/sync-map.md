# CLAUDART Sync Map

This file defines the shared sync contract between Claude Code and Codex.
Sync is snapshot-based: always read the current filesystem state from the named source side.
Do not use `git diff`, commit history, or session history to decide what changed.

## Base Template Authoring

This CLAUDART repository is the base template. Users copy `.claude/`, `.codex/`, `.agents/`, and `.claudart/` into their own projects.

In this base repository, `.claude` and `.codex/.agents` are peer source templates. Maintain both sides manually. Do not use sync to generate Codex from Claude or Claude from Codex while authoring the template.

Sync is for downstream user projects after installation, where one tool may have accumulated project-specific rule, command, skill, or agent changes that the other tool should receive.

## Invocation Semantics

- `sync claude` means: treat `.claude/` as the source of truth and update the Codex side.
- `sync codex` means: treat Codex files as the source of truth and update `.claude/`.
- `.claudart/CONTEXT.md` and `.claudart/JOURNAL.md` are shared state and are never copied, converted, or duplicated.

## Shared Memory

These files are canonical and tool-neutral:

- `.claudart/CONTEXT.md` - current declarative state, loaded by active agents.
- `.claudart/JOURNAL.md` - append-only history, read only on explicit history or learning tasks.

Sync commands MUST NOT create `.claude/CONTEXT.md`, `.claude/JOURNAL.md`, `.codex/CONTEXT.md`, or `.codex/JOURNAL.md`.

## Claude To Codex

When source is `claude`, update:

- `.claude/CLAUDE.md` -> `AGENTS.md` and `.codex/CODEX.md`
- `.claude/rules/*.md` -> `.codex/guidelines/*.md`
- `.claude/commands/*.md` -> `.codex/commands/*.md`
- `.claude/commands/*.md` -> `.agents/skills/<command-name>/SKILL.md` wrapper
- `.claude/agents/*.md` -> `.codex/agents/*.toml`

The Codex side must preserve Codex-native behavior:

- `AGENTS.md` stays concise and points Codex to `.codex/CODEX.md`, `.claudart/CONTEXT.md`, and `.codex/guidelines/*.md`.
- `.agents/skills/*/SKILL.md` wraps command specs for explicit `$skill-name` invocation.
- `.codex/agents/*.toml` stores Codex custom subagent definitions.
- `.codex/rules/*.rules` is reserved for Codex permission rules, not semantic coding guidelines.

## Codex To Claude

When source is `codex`, update:

- `AGENTS.md` and `.codex/CODEX.md` -> `.claude/CLAUDE.md`
- `.codex/guidelines/*.md` -> `.claude/rules/*.md`
- `.codex/commands/*.md` -> `.claude/commands/*.md`
- `.codex/agents/*.toml` -> `.claude/agents/*.md`

The Claude side must preserve Claude-native behavior:

- `.claude/CLAUDE.md` imports `@.claudart/CONTEXT.md`.
- `.claude/CLAUDE.md` never imports `@.claudart/JOURNAL.md`.
- `.claude/rules/*.md` use YAML frontmatter with `paths:` and `description:`.
- `.claude/agents/*.md` use YAML frontmatter with `name`, `description`, `tools`, `model`, and optional `memory`.

## Conflict Policy

- Source wins for generated target files.
- A target file is generated only if it contains `Generated by CLAUDART sync`.
- If a target file exists and does not contain that marker, report it as a manual target file and do not overwrite it without explicit user approval.
- Never delete target-only files automatically. Report them as orphan candidates.
- Preserve target-native local config files unless they are explicitly generated by sync.
- Base template files in this repository should not contain `Generated by CLAUDART sync` markers, except inside sync policy or command examples.

## Verification

Every sync run must check:

- No operational references to `.claude/CONTEXT.md`, `.claude/JOURNAL.md`, `.codex/CONTEXT.md`, or `.codex/JOURNAL.md` outside sync policy/command text.
- No operational `@.claudart/JOURNAL.md` import in `AGENTS.md`, `.claude/CLAUDE.md`, `.codex/CODEX.md`, `.claude/rules/`, or `.codex/guidelines/`.
- `.claudart/CONTEXT.md` exists and is at most 150 lines.
- `.claudart/JOURNAL.md` exists and remains append-only.
- Generated Codex skills each include a valid `name:` and `description:` frontmatter.
- Generated Codex agents each include `name`, `description`, `model`, and `sandbox_mode`.
